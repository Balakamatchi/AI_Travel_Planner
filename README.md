## Description

AI Travel Planner lets a logged-in user describe a trip — destination, budget, number of days, and traveler count — and get back a complete day-by-day itinerary, food recommendations, a packing checklist, an estimated budget breakdown, and a set of real places (attractions, hotels, restaurants) with coordinates, generated by **Google Gemini**. From there, users can save the trip, track actual expenses against the planned budget, view trip locations on an interactive Leaflet map, and log a personal travel journal with photos for each trip.

## Features

> Only features with a working, verified implementation in the codebase are listed.

### Authentication
- Register / Login / Logout with JWT
- Session persistence via `GET /api/auth/me` on app load
- Change password while logged in (requires current password)
- **Password reset without email** — verified by matching the account's registered name (`accountName`) to the email on file, then setting a new password directly. This is **not** a token-based or emailed reset link flow.

### Dashboard
- Total trips, upcoming trip count, total budget, total expenses, remaining budget, countries visited
- "Trips Created Over Time" bar chart and "Expenses by Category" pie chart (Recharts)
- Recent trips grid (latest 5)

### AI Trip Planner
- Generates a full itinerary via Google Gemini (`generate-trip` — preview only, not saved)
- Save the generated plan as a Trip (`generate-and-save`)
- Automatically fetches a real destination cover photo from the **Unsplash API** after generation
- Falls back to a deterministic mock itinerary generator if no Gemini API key is configured or the Gemini call fails, so the app remains demoable without external API access
- Rate-limited to 20 requests per 15 minutes per client to control API cost/abuse

### Trip Management
- Create trips manually or via AI
- List trips with search (by destination) and status filtering (upcoming / ongoing / completed / cancelled)
- View trip details: itinerary, food recommendations, packing checklist, travel dates
- Edit a trip's budget, start date, day count, and traveler count (end date and itinerary length are recalculated/trimmed accordingly on the frontend)
- Cancel a trip (soft status flag) or permanently delete it (cascades and deletes its expenses and journal entries)
- Trip status (`upcoming` / `ongoing` / `completed`) is **computed on the fly** from `startDate`/`endDate`, not stored — only `cancelled` is a stored override

### Budget Tracker
- Add, edit, and delete expenses per trip, categorized (accommodation, food, transport, activities, shopping, misc)
- Budget vs. spent bar chart per selected trip
- Remaining/over-budget indicator

### Interactive Map
- Renders trip places (attractions, hotels, restaurants) as markers on a Leaflet map
- Filter markers by place type
- Falls back to a "no map data" message when a trip has no valid coordinates

### Travel Journal
- Create journal entries per trip: title, content, 1–5 star rating, mood tag, and up to 6 uploaded photos
- Photos are stored on **Cloudinary** via Multer
- View, and delete journal entries (photos are cleaned up from Cloudinary on delete)

### Profile
- Update name, phone, and country
- Upload/replace avatar (stored on Cloudinary); falls back to a generated [ui-avatars.com](https://ui-avatars.com) initials avatar if none is set
- Change password

### UI / UX
- Light and dark mode (persisted to `localStorage`, toggled via a context provider)
- Glassmorphism card styling with Framer Motion entrance/hover animations
- Responsive sidebar navigation (collapsible on mobile) and top navbar
- Toast notifications (`react-hot-toast`) for success/error feedback across the app
- Landing page with feature highlights, aimed at logged-out visitors

## Installation Guide

### Prerequisites
- Node.js **v18+** (Node **22** confirmed working)
- MongoDB instance (local or MongoDB Atlas)
- A Cloudinary account (required — the server throws on startup without valid credentials)
- A Google Gemini API key (optional — falls back to a mock itinerary generator without it)
- An Unsplash API access key (optional — cover photos silently skip without it)

### Clone & Install

```bash
git clone <your-repo-url>
cd ai-travel-planner

# Install root dependency (concurrently, for running both servers together)
npm install

# Install backend dependencies
cd server
npm install

# Install frontend dependencies
cd ../client
npm install
```

## Environment Variables

Create a `.env` file inside **`server/`** with the following. Only variables actually read by the code are listed.

```env
# Server
PORT=5000
NODE_ENV=development
CLIENT_URL=http://localhost:5173

# MongoDB
MONGO_URI=your_mongodb_connection_string

# JWT
JWT_SECRET=your_jwt_secret
JWT_EXPIRE=7d
JWT_COOKIE_EXPIRE=7

# Cloudinary (required — server will not start without these)
CLOUDINARY_CLOUD_NAME=your_cloud_name
CLOUDINARY_API_KEY=your_api_key
CLOUDINARY_API_SECRET=your_api_secret

# Google Gemini (optional — falls back to mock itinerary if omitted)
GEMINI_API_KEY=your_gemini_api_key
GEMINI_MODEL=gemini-2.5-pro

# Unsplash (optional — cover photo fetch is skipped if omitted)
UNSPLASH_ACCESS_KEY=your_unsplash_access_key
```

Create a `.env` file inside **`client/`** (optional — defaults to `/api` for local dev with a proxy):

```env
VITE_API_URL=http://localhost:5000/api
```

## Authentication Flow

1. **Register / Login** — `authController.js` validates credentials, hashes passwords with bcrypt (`User.js` pre-save hook), and on success calls `sendTokenResponse()`, which:
   - Signs a JWT (`jsonwebtoken`, `JWT_SECRET`, default 7-day expiry)
   - Sets it as an **httpOnly cookie** (`secure` in production, `sameSite: lax`)
   - Also returns the token in the JSON response body
2. **Frontend storage** — `AuthContext.jsx` stores the returned token in `localStorage` and attaches it as a `Bearer` header on every request via an Axios request interceptor (`services/api.js`). This means the app relies on **both** the httpOnly cookie and a client-readable token — the backend's `protect` middleware accepts either.
3. **Protected routes** — `middleware/auth.js` reads the token (header first, then cookie), verifies it, loads the user, and attaches it to `req.user`. Missing/invalid/expired tokens return `401`.
4. **Frontend route guarding** — `ProtectedRoute.jsx` checks `AuthContext`'s `isAuthenticated` flag (populated via `GET /api/auth/me` on app load) and redirects unauthenticated users to `/login`.
5. **Global 401 handling** — the Axios response interceptor clears `localStorage` and redirects to `/login` on any `401` response.
6. **Password reset** — `ForgotPasswordPage.jsx` collects email + account name (full name) + new password, and `PUT /api/auth/reset-password` verifies the name matches the account on that email before allowing the password change. **No email is sent** — this is a self-service identity-check reset, not a token-based emailed link.
7. **Logout** — clears the httpOnly cookie server-side and clears `localStorage` client-side.

## Database Models

### User
`name`, `email` (unique), `password` (bcrypt-hashed, hidden by default), `avatar { url, publicId }`, `phone`, `country`, `theme` (`light`/`dark`), `resetPasswordToken`, `resetPasswordExpire` (defined on the schema but not currently used by the account-name-verification reset flow), timestamps.

### Trip
`user` (ref), `destination`, `country`, `coverImage`, `startDate`, `endDate`, `days`, `travelers`, `budget`, `status` (enum: `["cancelled"]` only — all other states are computed, not stored), `aiGenerated`, `itinerary` (array of day objects, each with an `activities` array — time, title, description, category, estimated cost), `foodRecommendations` (string array), `packingChecklist` (array of `{ item, packed }`), `bestTimeToVisit`, `estimatedBudgetBreakdown` (accommodation/food/transport/activities/misc), `places` (array of `{ name, type, description, lat, lng, rating }`), timestamps.

### Expense
`user` (ref), `trip` (ref), `title`, `amount`, `category` (enum: accommodation/food/transport/activities/shopping/misc), `date`, `notes`, timestamps.

### Journal
`user` (ref), `trip` (ref), `title`, `content`, `rating` (1–5), `mood` (enum: amazing/happy/neutral/tiring/disappointing), `photos` (array of `{ url, publicId }`), `entryDate`, timestamps.

## AI Integration

Itinerary generation uses **Google Gemini** via the `@google/genai` SDK (`server/utils/aiService.js`):

- A structured prompt requests a strict JSON shape: destination, country, best time to visit, budget breakdown, a day-by-day itinerary, food recommendations, a packing checklist, and a list of places with coordinates.
- The model is called with `responseMimeType: "application/json"` and the raw text response is parsed directly.
- **Fallback:** if no `GEMINI_API_KEY` is set, or the Gemini call throws/fails to parse, the server generates a deterministic **mock itinerary** locally (`generateMockPlan`) so the planner keeps working without any AI dependency — useful for demos and offline development.
- Category/place-type values returned by the model are sanitized server-side against the schema's allowed enums before saving, with unrecognized values coerced to `"other"` / `"attraction"`.
- **Cover photos are not generated by Gemini** (text generation cannot return real image URLs). After the itinerary is produced, a separate call to the **Unsplash Search API** fetches a real photo matching the destination name; if no Unsplash key is configured or the request fails, `coverImage` is left empty and the frontend displays a placeholder instead.

## Cloudinary Integration

Cloudinary is used for **two** upload paths, both via `multer-storage-cloudinary`:

1. **Avatar uploads** (`PUT /api/users/avatar`) — stored under `ai-travel-planner/avatars`, replacing (and deleting) any previous avatar.
2. **Journal photo uploads** (`POST`/`PUT /api/journals`) — up to 6 images per entry, stored under `ai-travel-planner/journal-photos`; deleted from Cloudinary when a journal entry is removed.

Both are limited to image MIME types, with file size caps (3MB avatars, 5MB journal photos) and automatic resizing (max width 1600px).

> Note: a `uploadTripCover` upload configuration exists in `middleware/upload.js` for a trip cover-image upload feature, but it is **not wired to any route** and there is no corresponding UI — trip cover images currently come only from the automatic Unsplash fetch during AI generation, not manual upload.

## Map Integration

The **Map View** page (`MapViewPage.jsx`) uses **React Leaflet** with OpenStreetMap tiles to plot a selected trip's `places` array as markers, color-coded by type (attraction / hotel / restaurant). Users can filter which place types are shown, and marker popups display the place name, type, description, and rating. If a trip has no places with valid coordinates, the map shows an empty-state message instead.

## Responsive Design

Built with Tailwind CSS using a mobile-first approach:
- Grid layouts collapse from multi-column (`lg:grid-cols-3/4`) down to single/dual-column on small screens across the Dashboard, Trips, Budget, and Map pages.
- The sidebar is an off-canvas drawer on mobile (toggled via a hamburger menu in the navbar) and a static column on large screens.
- Dark mode is available throughout via a `class`-based Tailwind strategy, toggled from the navbar/landing page and persisted in `localStorage`.

## Security Features

- Passwords hashed with **bcrypt** (10 salt rounds) before storage; password field excluded from queries by default (`select: false`)
- **JWT-based authentication**, verified on every protected route via middleware
- Auth token delivered as an **httpOnly cookie** (`secure` flag enabled in production, `sameSite: lax`) in addition to the response body
- **CORS** restricted to a configured `CLIENT_URL` origin with credentials enabled
- **express-validator** input validation on registration, login, password change, trip creation, and expense/journal creation
- **Rate limiting** on AI generation endpoints (20 requests / 15 minutes) to prevent abuse of the paid Gemini API
- **Centralized error handling** middleware that normalizes Mongoose `CastError`, duplicate-key, `ValidationError`, and JWT errors into consistent JSON responses, hiding stack traces in production
- Every data-fetching operation is scoped to `req.user._id`, preventing users from accessing other users' trips, expenses, or journal entries
- File uploads restricted to image MIME types with explicit size limits
